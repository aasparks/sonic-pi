Sonic Pi: WORK IN PROGRESS

John Clements <clements@racket-lang.org>

This package is a collection of functions and a very primitive language level that follows the lead of Sonic Pi. Specifically, (like Sonic Pi) it uses scsynth as its sound generation engine, and creates the same network of units that Sonic Pi does.

This is very much a work in progress. I’m releasing it as a project so that others can try it out and steal parts of it.

Want to try it out? Install the package, then open one of the examples in "examples/" in DrRacket and click Run.

Tell me what happens!

1 Prerequisites

SuperCollider must be installed on the system prior to use. For Windows users, it must be installed to "C:\\Program Files"

2 An Imperative Interface

2.1 SuperCollider Abstraction

 (require sonic-pi/scsynth/scsynth-abstraction)
  package: sonic-pi
One thing that I don’t like that much about Sonic Pi is its unabashedly imperative interface.
However, underneath what I hope will be a nice clean language (lsonic), I’m also going to publish the lower-level imperative interface. For one thing, it’s pretty obvious what it’ll look like: it’ll look like Sonic Pi. That means less design work. Here are some functions:

procedure

(startup) → context?

Starts an scsynth, returns a context that can be used to ... well, to create a job context.

procedure

(start-job ctxt) → job-context?

  ctxt : context?
Given a context, starts a job, and return a handle for that job. A job corresponds to a “piece of music.”

procedure

(end-job job-ctxt) → void?

  job-ctxt : job-context?
Ends a job: fades out the job and cancels all scheduled notes associated with the job.

procedure

(play-synth job-ctxt name time param-part ...) → void?

  job-ctxt : job-context?
  name : string?
  time : inexact-real?
  param-part : (or string? real?)
Play note, sample, or fx using job-ctxt, at time time, specified in inexact milliseconds since the epoch. If the time value is less than the current number of milliseconds since the epoch, the note will be played immediately. (Note that this means that you can always specify 0.0 to play a note immediately.)

2.2 Note

 (require sonic-pi/note) package: sonic-pi

procedure

(note synth-name pitch param-part ...) → note?

  synth-name : string?
  pitch : (or string? real?)
  param-part : (or string? real?)
Creates a note. In addition to the synth-name and pitch, users may specify non-default values for one of many other parameters using and interleaved parameter-name / value style. For instance:

(note "saw" 78 "attack" 0.5 "amp" 0.5)

The pitch may be specified as a MIDI note number, or in musical notation. Musical notation may include the note, accidental, or octave in that order :
(note "bronky" 39)
(note "bronky" "A")
(note "pulse" "C2")
(note "beep" "af3")
Here’s the full list of parameters and their defaults:
'((note_slide 0)
  (note_slide_shape 5)
  (node_slide_curve 0)
  (amp 1)
  (amp_slide 0)
  (amp_slide_shape 5)
  (pan 0)
  (pan_slide 0)
  (pan_slide_shape 5)
  (pan_slide_curve 0)
  (attack 0)
  (decay 0)
  (sustain 0)
  (release 1)
  (attack_level 1)
  (sustain_level 1)
  (env_curve 2)
  (out_bus 12.0))

procedure

(chord synth-name pitch interval) → (listof note?)

  synth-name : string?
  pitch : (or string? real?)
  interval : string?
creates a list of notes in the specified chord. For example:

(chord "saw" "a" "minor")

Here is a list of all supported intervals:

'("major" "minor" "major7" "dom7" "minor7" "aug" "dim" "dim7")

procedure

(control-note note param-part ...) → note?

  note : note?
  param-part : (or string? real?)
Reassigns the optional paramaters to the specified paramaters, using the note’s original values as the default.

procedure

(note? note) → boolean?

  note : any/c
returns true for notes.

2.4 FX

 (require sonic-pi/fx) package: sonic-pi

procedure

(fx name param-part ... block) → fx?

  name : string?
  param-part : (or string? real?)
  block : (-> any/c)
Creates a sound effect to be applied to other sounds. name may be one of the predefined sound effects provided by the original Sonic Pi. Unlike with notes, all paramaters provided are passed to SuperCollider, not just the defaults. The last param-part is expected to be a block, containing a set of sounds to be played with this sound effect. For instance:

(fx "bitcrusher"
    "bits" 8
    (λ () (synth "beep" 60)))

procedure

(control-fx f param-part ...) → fx?

  f : fx?
  param-part : (or string? real?)
Reassigns a sound effect with new parameters. For instance:
(define my-fx (fx "bitcrusher" "bits" 8
                  (λ () (synth "beep" 60))))
(control-fx my-fx "bits" 2)

procedure

(fx? fx) → boolean?

  fx : any/c
returns true for fx’s

procedure

(set-fx-busses fx in-bus out-bus) → fx?

  fx : fx?
  in-bus : integer
  out-bus : integer?
Sets the in-bus and out-bus parameters for a sound effect

procedure

(set-block fx block) → fx?

  fx : fx?
  block : (-> any/c)
Sets the block of music to be played for a sound effect

3 A Live Coding Environment

This package provides a primitive language similar to Sonic Pi with a simple live coding environment within DrRacket. The provided play and stop buttons will update the playing music in real time and stop the music and shutdown SuperCollider, respectively.

3.1 Sounds

 #lang sonic-pi/lsonic package: sonic-pi

procedure

(synth synth-name pitch param-part ...) → note?

  synth-name : string?
  pitch : (or string? real?)
  param-part : (or string? real?)
Plays a note of the given pitch using the given synth-name. This is unlike Sonic Pi, which does not require you to input a synth each time. Users may specify non-default values for one of many other parameters using and interleaved parameter-name / value style. For instance:

(synth "saw" 78 "attack" 0.5 "amp" 0.5)

The pitch may be specified as a MIDI note number, or in musical notation. Musical notation may include the note, accidental, and octave :
(synth "bronky" 39)
(synth "bronky" "A")
(synth "pulse" "C2")
(synth "beep" "af3")
Here’s the full list of parameters and their defaults:

  (node_slide_curve 0)
  (amp 1)
  (amp_slide 0)
  (amp_slide_shape 5)
  (pan 0)
  (pan_slide 0)
  (pan_slide_shape 5)
  (pan_slide_curve 0)
  (attack 0)
  (decay 0)
  (sustain 0)
  (release 1)
  (attack_level 1)
  (sustain_level 1)

procedure

(chord synth-name pitch interval) → (listof note?)

  synth-name : string?
  pitch : (or string? real?)
  interval : string?
Play a chord. For example:

(chord "saw" "a" "minor")

Here is a list of all supported intervals:

'("major" "minor" "major7" "dom7" "minor7" "aug" "dim" "dim7")

procedure

(psleep time) → void

  time : inexact-real?
Sleep for time seconds before playing more. The following two beeps will happen at the same time
(synth "beep" 60)
(synth "beep" 62)
Whereas the next two will happen one after the other
(synth "beep" 60)
(psleep 1)
(synth "beep" 62)

procedure

(sample name param-part ...) → sample?

  name : string?
  param-part : (or string? real?)
Play a sample. These are the same samples provided by Sonic Pi. To sample a local file, set the name to the full file path of the sample The optional arguments to sample do not include all the same as the original Sonic Pi. Here is a list of the supported parameters:
'((amp 1)
  (amp_slide 0)
  (amp_slide_shape 1)
  (amp_slide_curve 0)
  (attack 0.0)
  (decay 0)
  (sustain -1)
  (release 0.0)
  (attack_level 1)
  (decay_level -1)
  (sustain_level 1)
  (env_curve 1)
  (pan 0)
  (pan_slide 0)
  (pan_slide_shape 1)
  (pan_slide_curve 0)
  (lpf -1)
  (lpf_slide 0)
  (lpf_slide_shape 1)
  (lpf_slide_curve 0)
  (hpf -1)
  (hpf_slide 0)
  (hpf_slide_shape 1)
  (hpf_slide_curve 0)
  (rate 1))

procedure

(block sounds) → (-> any/c)

  sounds : any/c
Creates a block of sound(s) similar to the do ... end blocks of Sonic Pi. This is accomplished by wrapping the set of events in a closure. Thus

(block (synth "pretty_bell" 60))

is equivalent to

(λ () (synth "pretty_bell" 60))

blocks are used when applying fx, looping, or threading. For example:
(live_loop "foo"
           (block (sample "bd_boom")
                  (psleep 2)))

procedure

(fx name param-part ... block) → fx?

  name : string?
  param-part : (or string? real?)
  block : (-> any/c)
Play block of sounds using the specified sound effect. The effects provided are the same as Sonic Pi. Each fx has a different set of parameters for it, so the defaults are determined by the synth definition. All params given will be passed and any that don’t apply to the fx will be ignored. For example:
(fx "reverb" "room" 2
    (block (synth "saw" "bf")))

procedure

(thread_s block) → thread_s?

  block : (-> any/c)
Plays the block in a separate thread from the rest of the music. This does not actually spawn a new thread of execution, but simulates threading by scheduling events simultaneously. Use to overlay sounds:
(thread_s (block
           (fx "reverb" (block
                         (loop 16 (block
                                   (sample "ambi_choir" "rate" (choose 0.5
                                                                       (/ 1.0 3)
                                                                       (/ 3.0 5))
                                           "pan" (rrand -1 1))
                                   (psleep 0.5)))))))
 
 
(fx "wobble" "phase" 2
    (block (fx "echo" "mix" 0.6
               (block (loop 12
                            (block
                             (sample "drum_heavy_kick")
                             (sample "bass_hit_c" "rate" 0.8 "amp" 0.4)
                             (psleep 1)))))))

procedure

(loop reps block) → loop?

  reps : integer?
  block : (-> any/c)
Loop the block reps times. To loop infinitely, provide a large number for the reps. Loops are blocking, so use in a thread to overlay loops.

procedure

(choose elem ...) → any/c

  elem : any/c
Randomly pick one element from the ones provided. If any element passed to it is a list, it will behave like choose-list. For example:
(choose (sample "bd_haus")
        (sample "bd_boom")
        (sample "elec_plip"))

procedure

(choose-list elements) → any/c

  elements : (listof any/c)
Same as choose but for elements specifically in a list, though choose will accept lists as well.

(choose-list (chord "beep" "a" "minor"))

procedure

(rrand min max) → real?

  min : real?
  max : real?
Generates a random number between min and max.

procedure

(rrand_i min max) → integer?

  min : integer?
  max : integer?
Generates a random integer from min to max, inclusive.

3.2 Live Coding

Live coding is implemented in a very simple fashion for use with DrRacket. There are two button provided with this package: stop and play.

When using the live coding environment, first press the DrRacket run button. This will start up SuperCollider and begin playing your music. After making changes to the source, press the provided play button to update the music while it plays. It may take two or three iterations of a loop to update. When you wish to end the music that is playing, press the stop button. This will also shut down SuperCollider, so you will need to re-run the program to start it back up. The stop button will always attempt to shut down SuperCollider so this can be used if it failed to shut down previously.

procedure

(live_loop name block) → live_loop?

  name : string?
  block : (-> any/c)
Play a block of music in an infinite loop with the ability to update. Live loops are executed in their own threads so they must have unique names and they are non-blocking. The following example comes from the Sonic Pi tutorial:

(live_loop "boom"
          (block
           (fx "reverb" "room" 1
               (block (sample "bd_boom"
                              "amp" 10
                              "rate" 1)
                      (sample "elec_blip"
                              "amp" 10
                              "rate" 1)
                      (psleep 8)))))
(live_loop "guit"
           (block (fx "echo" "mix" 0.3 "phase" 0.25
                      (block (sample "guit_em9" "rate" 0.5)
                             (sample "guit_e_fifths" "rate" 0.5)
                             (psleep 8)))))
After running, comment as you see fit, and press the play button to hear it change. It may take a few loops for the changes to take effect.

4 Bugs

There are bound to be plenty of bugs, so report!

Here is a list of known bugs:

  • Loops may fall out of sync or skip an iteration due to timing delays

  • Bitcrusher fx may not work in Windows
